## 1. 오픈크롤봇 완전정복

오픈크롤봇(Moltbot)의 모든 기능과 사용법을 설명하는 완전 가이드입니다.

### 1.1 개요

오픈크롤봇은 WhatsApp, Telegram, Discord, iMessage 등 다양한 채널을 지원하는 개인형 AI 어시스턴트입니다. GitHub로 오픈소스로 개발되며, 온보딩 위저드를 통해 누구나 쉽게 시작할 수 있습니다.

### 1.2 주요 특징

- **멀티 채널 지원**: WhatsApp, Telegram, Discord, iMessage, Mattermost
- **멀티 에이전트**: 여러 전문 에이전트를 동시에 운용 가능
- **A2A 통신**: 에이전트 간 직접 메시지 전송 가능
- **스킬 시스템**: 기능 확장을 위한 모듈형 스킬 구조
- **RAG 통합**: 외부 지식베이스 연동 가능
- **그룹 채팅 지원**: 텔레그램 그룹에서 @멘션으로 활성화

### 1.3 아키텍처

```
사용자
    ↓
WhatsApp / Telegram / Discord
    ↓
┌─────────────────────────┐
│  OpenClaw Gateway      │
│  (ws://127.0.0.1:18789)  │
└───────────────────────────┘
    ↓           ↓           ↓
┌─────────────────────────────┐
│   Agent A   Agent B   Agent C  │
└─────────────────────────────┘
    ↓           ↓           ↓
  Skills (RAG, Web, File)   Memory (Shared)
```

## 2. 활용 팁 & 베스트 프랙티스

파워유저들을 위한 50가지 활용 팁을 정리했습니다.

### 2.1 초기 설정 (1-10)

1. **`openclaw onboard` 실행**: 모든 세팅 자동화
2. **서비스 데몬 등록**: 부팅 시 자동 시작
3. **`openclaw channels login`**: 채널 연동 QR 코드 스캔
4. **로컬 대시보드**: `http://127.0.0.1:18789/` 접속
5. **AGENTS.md 작성**: 에이전트 역할 명확히 정의
6. **SOUL.md 작성**: 에이전트 성격/말투 설정
7. **워크스페이스 키 관리**: 환경 변수로 API 키 보호

### 2.2 채널 설정 최적화 (11-15)

11. **`allowFrom` 화이트리스트**: 허용된 연락처만 응답
12. **그룹 채팅 `requireMention: true`**: 멘션 시에만 응답
13. **`mentionPatterns` 커스텀**: 여러 멘션 패턴 등록
14. **채널별 에이전트 라우팅**: 채널마다 다른 에이전트 할당
15. **`activation: always|mention`**: 그룹 활성화 모드 선택

### 2.3 프롬프트 엔지니어링 (21-30)

21. **역할 명확히 정의**: "당신은 ~입니다"로 시작하여 에이전트 성격 부여
22. **Few-shot 예시 추가**: 2-3개 이상적인 응답 예시
23. **구조화된 출력 요청**: 마크다운, JSON 등 형식 지정
24. **출력 길이 제한**: "3문장 이내로" 등 지시
25. **단계별 지시**: "먼저 생각하고, 그다음 답변하세요"
26. **도구 호출 가이드**: "검색은 web_search 툴을 사용하세요" 등
27. **부정적인 응답 처리**: "모르는 건은 '확인 필요'라고 답변"
28. **톤 조절**: "간결한 응답, 전문적인 답변은 다른 모델로"
29. **NO_REPLY 사용**: 리플리가 필요 없을 때 전용 태그 사용
30. **맥락 제거**: 불필요한 설명 제거, 핵심만 유지

### 2.4 성능 최적화 (31-40)

31. **모델 라우팅**: 간단 작업은 저비용 모델, 복잡한 작업은 고비용 모델
32. **캐싱 활성화**: 반복 쿼리는 캐시 히트 응답
33. **배칭 처리**: 유사한 요청을 묶어서 한 번에 API 호출
34. **출력 간결화**: 장문없이 핵심만 답변
35. **컨텍스트 윈도우 최적화**: 불필요한 대화 기록 삭제, contextTokens 조정
36. **스트리밍 활용**: 긴 응답 청크로 전달, 사용자 경험 개선
37. **툴 응답 최적화**: 필요한 툴만 호출, 불필요한 API 호출 회피
38. **비동기 처리**: 병렬로 가능한 작업은 비동기로 처리
39. **세션 풀링**: 오래된 세션 정기적으로 정리, 메모리 확보
40. **요약 기능 활용**: 긴 대화 내용을 요약하여 토큰 절약

### 2.5 비용 절감 (41-50)

41. **API 호출 최소화**: 불필요한 도구 호출 회피, 토큰 효율적 사용
42. **일일/월간 예산 설정**: `budget.dailyLimit`, `budget.monthlyLimit` 설정
43. **비용 모니터링**: 토큰 사용량 추적, 예산 초과 알림
44. **저비용 모델 우선**: GPT-4o-mini, Gemini Flash 등 저비용 모델을 우선
45. **출력 토큰 감소**: 간결한 응답 유도, 긴 설명 회피
46. **캐시 TTL 설정**: 반복 쿼리는 요청은 캐시로 응답
47. **배칭으로 30-50% 절감**: 유사한 쿼리를 묶어서 처리
48. **과금 플랜 사용 가능**: n8n 무료 플랜 활용, Make Starter 사용
49. **쿨드 타임아웃**: API 요청 타임아웃 설정으로 불필요한 대기 방지
50. **평균 토큰 사용량 모니터링**: 사용 패턴 분석, 비효율적인 요청 식별

### 2.6 보안 & 프라이버시 (51-54)

51. **API 키 환경변수 저장**: config 파일에 직접 저장 금지
52. **API 키 `.env` 파일 관리**: 환경 변수로부터 로드, `.gitignore`에 추가
53. **`allowFrom` 화이트리스트**: 승인된 연락처만 허용
54. **민감 정보 필터링**: 신용카드, 비밀번호 등 자동 마스킹 후 전달
55. **프롬프트 인젝션 방지**: 사용자 입력 검증, 위험한 명령어 차단
56. **읽기 전용 경로**: 파일 읽기 경로를 제한, 상위 디렉토리 접근 차단
57. **쓰기 전용 경로**: 파일 쓰기 경로를 제한, 데이터 보호
58. **`requireMention` 활성화**: 그룹에서 멘션 없이 자동 응답 차단
59. **세션/로그 보호**: 개인정보 포함된 로그 암호화 또는 삭제
60. **업데이트 자동화**: 보안 패치를 적용하기 위한 자동 업데이트 설정

### 2.7 워크플로우 자동화 (61-64)

61. **Cron Job 설정**: 정기적 작업 자동화 (백업, 보고, 리마인더)
62. **Webhook 통합**: 외부 서비스와의 데이터 자동 전송
63. **Gmail Pub/Sub**: 이메일 트리거로 자동 작업 시작
64. **n8n 통합**: 워크플로우 자동화, 서비스 간 연결
65. **Make (Integromat) 통합**: 복잡한 자동화 시나리오
66. **GitHub Actions**: CI/CD 파이프라인, 코드 테스트 자동화
67. **IFTTT/Zapier**: 간단한 작업 자동화, 서비스 연결

### 2.8 유지보수 (65-74)

## 3. 멀티에이전트 아키텍처

오픈크롤봇에서의 멀티에이전트 시스템을 구축하기 위한 패턴과 방법을 정리했습니다.

### 3.1 통신 방법 비교

| 방식 | 장점 | 단점 | 추천 상황 |
|------|--------|--------|-----------|
| 멘션 기반 | 자연스러운 사용, 사용자 제어 | 호출 패턴 필요 | 그룹 채팅, 일상 대화 |
| 웹훅 | 비동기 처리, 실시간성 | 구현 복잡도 높음 | 서버 간 통신, 복잡한 워크플로우 |
| 공유 메모리 | 상태 일관성, 장기 보관 | 인프라 비용, 관리 오버헤드 | 상태 공유 필요 |
| A2A 통신 | 팀 네이티브, sessions_send | 세션 분리 필요 | OpenClaw 내장 통신 |
| MCP | 표준화된 툴 접근 | 제공자 의존 | 외부 도구 통합 |

### 3.2 오케스트레이터 패턴

| 패턴 | 설명 | 구조 | 장점 | 단점 |
|------|------|--------|--------|--------|
| 중앙 집중형 | 단일 오케스트레이터가 조율 | 계층형 | 빠른 의사결정, 확장성 제한적 | SPOF 위험 |
| 계층형 | 계층별 상태 관리 | 상위 에이전트 → 하위 에이전트 | 확장성 좋음, 유연성 높음 | 계층 구조 복잡 |
| 분산형 | 에이전트 자율적 협력 | 피어 투피어 통신 | 내결성 강화, SPOF 해소 | 조정 어려움 |
| 하이브리드 | 중앙 조율 + 분산 실행 | 유연성 + 안정성 | 구현 복잡도 최고 |

### 3.3 역할 분담 최적 구조

| 역할 | 책임 | 추천 모델 | 핵심 기술 |
|------|------|-----------|----------|
| 오케스트레이터 (세나) | 작업 분배, 최종 승인, 팀 조율 | Claude Opus / GPT-5 | 워크플로우 설계 |
| 리서처 (미루) | 데이터 수집, 분석, 근거 기반 의견 | GLM-4.7 / Gemini Pro | 검색, RAG |
| 빌더 (하나) | 문서 생성, 코드 작성, 빌드 | Claude Sonnet / GPT-4o | 프롬프트 엔지니어링 |
| 리뷰어 (유리) | 비평, 품질 검증, 피드백 | Claude Opus / GPT-5 | 평가 기준, 테스트 |
| 전문가 | 도메인별 전문 지식 | Claude Opus / GPT-5 | 전문 데이터베이스 연동 |

### 3.4 실제 구현 사례

**AutoGen Python Example**
```python
from autogen import AssistantAgent, UserProxyAgent, GroupChat

researcher = AssistantAgent("researcher", llm_config=llm_config)
writer = AssistantAgent("writer", llm_config=llm_config)
user = UserProxyAgent("user")

group_chat = GroupChat(
    agents=[researcher, writer, user],
    messages=[],
    max_round=10
)

result = group_chat.run("오픈크롤봇 분석해줘")
```

**CrewAI Python Example**
```python
from crewai import Agent, Task, Crew

researcher = Agent(
    role='리서처',
    goal='데이터 수집 및 분석',
    backstory='호기심 많은 데이터 수집 전문가',
    tools=[search_tool, data_tool]
)

writer = Agent(
    role='작성자',
    goal='문서 작성',
    tools=[writer_tool]
)

task1 = Task(description='주제 조사', agent=researcher)
task2 = Task(description='보고서 작성', agent=writer)

crew = Crew(
    agents=[researcher, writer],
    tasks=[task1, task2],
    process=Process.sequential
)

result = crew.kickoff()
```

**LangGraph Python Example**
```python
from langgraph.graph import StateGraph, MessagesState
from langchain_core.prompts import ChatPromptTemplate

class TeamState(MessagesState):
    research_results: str

    final: str

def research_node(state):
    # 리서치 수행
    return {"research_results": "검색 결과"}

def writer_node(state):
    # 문서 작성
    return {"draft": "문서 초안"}

def review_node(state):
    # 비평
    return {"final": "최종 문서"}

workflow = StateGraph(TeamState)
workflow.add_node("researcher", research_node)
workflow.add_node("writer", writer_node)
workflow.add_edge("reviewer", review_node)
```

### 4.4 일반적인 실수 & 해결책

| 실수 | 증상 | 해결책 |
|------|------|----------|
| ContextTokens 너무 높게 설정 | 비용 과다, 느린 응답 | 간단 작업은 32K-64K, 복잡한 리서치만 128K+ |
| Temperature 너무 높게 설정 | 응답 일관성 없음, 반복 질문 | 기본 0.3-0.7, 창의적 작업만 0.8+ |
| 스킬 제한 없음 | 비용 폭증, 오류 반복 | 스킬별 호출 제한 설정 |
| 로그/세션 누적 | 느려짐, 디스크 공간 부족 | 주기적 세션 정리, 로그 순환 |
| A2A 통신 활성화 안 함 | 팀 협업 불가 | sessions_send 권한 부여 |
| 모델 라우팅 미사용 | 간단 작업에도 비싼 모델 사용 | 작업 유형별 모델 분배 |
| 캐시 비활성화 | 반복 쿼리로 비용 낭비 | 캐시 활성화, TTL 1h 설정 |
| 배칭 미사용 | 유사 쿼리 개별 처리 | 배칭으로 20-30% 절감 |

## 6. 스킬빌더 스킬 설계

오픈크롤봇에서 스킬을 만들고 관리하기 위한 시스템을 설계합니다.

### 6.1 스킬 구조

```
┌─────────────────────────────────┐
│           스킬 (Skill)        │
│  ┌───────────────────────┐  │
│  │    프롬프트 (Prompt)    │  │
│  │    (시스템 + 사용자)        │  │
│  └───────────────────────┘  │
│  ┌───────────────────────┐  │
│  │   워크플로우 (Workflow)     │  │
│  │   - 단계별 절차            │  │
│  │   - 조건 분기               │  │
│  │   - 도구 호출 순서             │  │
│  └───────────────────────┘  │
│  ┌───────────────────────┐  │
│  │       툴 (Tools)          │  │
│  │   - 외부 API                 │  │
│  │   - 내장 함수                 │  │
│  │   - 파일 조작                 │  │
│  └───────────────────────┘  │
└───────────────────────────────────┘
```

### 6.2 TDD 방식 (테스트 주도 개발)

```
1. RED: 테스트 실패 (테스트 케이스 작성)
         ↓
2. GREEN: 테스트 통과 (최소 기능 구현)
         ↓
3. REFACTOR: 코드 개선 (중복 제거, 최적화)
         ↓
4. (반복)
```

### 6.3 버전 관리

```
v1.0.0 → v1.1.0 → v1.2.0 → v2.0.0
Major: 하위 호환성 깨짐
Minor: 기능 추가, 하위 호환
Patch: 버그 수정, 하위 호환
```

### 6.4 스킬 생성 워크플로우

1. **초기화**: `openclaw skill create`
2. **프롬프트 설계**: prompt.md 작성
3. **워크플로우 정의**: workflow.md 작성
4. **툴 구현**: tools.py 작성
5. **테스트 작성**: tests/ 디렉토리 생성
6. **검증**: `openclaw skill test`
7. **배포**: `openclaw skill publish`
