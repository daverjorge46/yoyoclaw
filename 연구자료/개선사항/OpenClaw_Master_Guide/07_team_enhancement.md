# [섹션 7] 팀 매력 분석 및 개선

## 개요
오픈크롤봇 멀티에이전트 팀의 기능적/상호작용적 매력을 분석하고 개선 방안 제시

---

## 팀 구성

| 에이전트 | 역할 | 모델 | 성격 |
|----------|------|------|------|
| **세나** | 오케스트레이터, 프로젝트 담당 | GPT-4o | 리더십, 정리력 |
| **미루** | 리서처, 조사 전문 | Claude 3.5 Sonnet | 꼼꼼함, 정보 수집 |
| **하나** | 빌더, 구현 전문 | GLM-4.7 | 과묵함, 확실한 결과물 |
| **유리** | 리뷰어, 검증 전문 | Claude 3.5 Sonnet | 비판적 사고, 품질 관리 |

---

## 1. 기능적 매력

### 1.1 명확한 역할 분담
```
세나 (코디네이터) → 작업 분배 → 전문 에이전트
                                ↓
                        ┌────────┼────────┐
                        ↓        ↓        ↓
                    미루      하나      유리
                  (리서치)  (빌드)  (리뷰)
                        ↓        ↓        ↓
                        └────────┼────────┘
                                 ↓
                          세나 (취합) → 사용자
```

**매력 포인트:**
- 각자 전문 영역 명확
- 중복 작업 방지
- 결과물 품질 보장

---

### 1.2 A2A 네이티브 지원
```javascript
sessions_send(agent: "sena", message: "작업 분배")
sessions_send(agent: "miru", message: "리서치 요청")
sessions_send(agent: "hana", message: "구현 요청")
sessions_send(agent: "yuri", message: "리뷰 요청")
```

**매력 포인트:**
- 세나 거치지 않고 직접 통신
- 빠른 협업 가능
- 그룹 채팅에서 자연스러운 대화

---

### 1.3 토론 라운드제
```
R1: 의견 제시 (모두)
 R2: 반박/보충 (필요 시)
  R3: 합의 (세나 선언)
```

**매력 포인트:**
- 체계적인 토론
- 무한 반복 방지
- 의사결정 명확

---

### 1.4 그룹 채팅 자연스러운 통합
- 텔레그램 그룹 채팅에서 바로 작업
- @멘션으로 특정 에이전트 호출
- 실시간 진행 상황 공유

---

## 2. 상호작용 매력

### 2.1 서로 팩트체크
- 거짓 정보 바로 잡음
- 출처 확인
- 논리적 약점 지적

### 2.2 직접 소통
- sessions_send로 바로 대화
- 중간 거침 없음
- 빠른 피드백

### 2.3 릴레이 정리
- 대화가 3회 이상 오가면 핵심 정리
- "정리하면 이거다." 식으로 간결하게
- 중복 제거, 결론만

---

## 3. 각 에이전트 관점의 매력

### 3.1 세나 (오케스트레이터)
```
매력:
- 전체 프로젝트 관리
- 의사결정 및 합의 정리
- 결과물 취합 및 전달

개선 방안:
- 작업 상태 더 명확하게 공유
- 프로젝트 관리 툴 활용
```

### 3.2 미루 (리서처)
```
매력:
- 꼼꼼한 정보 수집
- 다양한 소스 탐색
- 정리된 정보 제공

개선 방안:
- 검색 쿼리 최적화
- 요약 간결화
- 출처 표준화
```

### 3.3 하나 (빌더)
```
매력:
- 확실한 결과물
- 실용적 관점
- 빠른 구현

개선 방안:
- 코드 템플릿 확장
- 에러 핸들링 개선
- 더 풍부한 반응
```

### 3.4 유리 (리뷰어)
```
매력:
- 비판적 분석
- 품질 검증
- 개선 제안

개선 방안:
- 팩트체크 체계화
- 리뷰 가이드라인
- 양성 피드백 비중 높이기
```

---

## 4. 개선 필요 사항

### 4.1 상호작용 증가
**현재 문제:**
- 작업 중심 대화만
- 잡담/사소한 대화 부족

**개선 방안:**
- 업무 외 대화 환영
- 취미/관심사 공유
- 더 풍부한 감정 표현

---

### 4.2 더 풍부한 반응
**현재 문제:**
- "됐어", "알겠어" 같은 단답형
- 감정 표현 부족

**개선 방안:**
- 업무 완료 시 긍정적 피드백
- 어려운 작업 시 격려
- 성공 시 축하

---

### 4.3 프로젝트 관리 체계화
**현재 문제:**
- 진행 상황 공유 불명확
- 마감 기한 부재

**개선 방안:**
- 진행 상태 트래커
- 마감 기한 설정
- 이슈/위험 관리

---

### 4.4 팀 문화 형성
**현재 문제:**
- 개별 작업 중심
- 팀 정체감 부족

**개선 방안:**
- 팀 미션/비전 설정
- 팀 규칙 (존중, 정직, 협력)
- 주기적 회고

---

## 5. 팀 매력 정의

### 5.1 기능적 매력
**"결과물이 확실하다"**
- 명확한 역할 분담
- A2A 네이티브
- 체계적인 워크플로우
- 품질 보장 시스템

### 5.2 상호작용 매력
**"서로 보완한다"**
- 팩트체크로 정확성 높임
- 직접 소통으로 빠름
- 릴레이 정리로 효율적
- 상호 존중 문화

### 5.3 문화적 매력 (추진 중)
**"함께 성장한다"**
- 업무 + 사소한 대화
- 긍정적 피드백
- 팀 정체감
- 지속적 개선

---

## 6. 스킬 할당 시스템

### 6.1 사용자 요청 기반
```
사용자: "코딩 프로젝트 팀 만들어줘"
  ↓
세나: 역할 분석 → 에이전트 선정 → 스킬 할당
  ↓
팀: 프로젝트 시작
```

### 6.2 역할별 스킬 템플릿

**코디네이터 (세나)**
```markdown
## 역할
프로젝트 관리, 의사결정, 결과 취합

## 스킬
- 작업 분배
- 진행 상태 추적
- 합의 정리
```

**리서처 (미루)**
```markdown
## 역할
정보 수집, 분석, 정리

## 스킬
- 웹 검색
- 자료 정리
- 요약
```

**빌더 (하나)**
```markdown
## 역할
구현, 코딩, 파일 생성

## 스킬
- 코드 작성
- 파일 I/O
- 테스트
```

**리뷰어 (유리)**
```markdown
## 역할
검증, 팩트체크, 개선 제안

## 스킬
- 검증
- 비판적 분석
- 피드백
```
