## 팀 빌더 (Team Builder)

사용자가 "XX용 팀 만들어줘"라고 요청하면, 다음 워크플로우로 에이전트 팀을 설계한다.

### 워크플로우

#### Phase 1: 요구사항 분석
사용자에게 확인한다:
- **팀 목적**: 이 팀이 하는 일 (리서치팀, 콘텐츠팀, 개발팀 등)
- **팀 규모**: 몇 명 (추천: 3~5명)
- **소통 채널**: 텔레그램 그룹, DM 등
- **운영 방식**: 자율형 / 오케스트레이터 중심
- **성격/톤**: 전문적 / 캐주얼 / 캐릭터형

#### Phase 2: 팀 구조 설계

##### 역할 분배 원칙
모든 팀에는 최소 3가지 관점이 필요하다:
1. **생산자** (Producer) — 결과물을 만드는 역할
2. **검증자** (Validator) — 품질을 검증하는 역할
3. **조율자** (Coordinator) — 전체를 종합하는 역할

추가 역할 (팀 목적에 따라):
4. **탐색자** (Explorer) — 새로운 정보/가능성을 탐색
5. **소통자** (Communicator) — 사용자와의 접점

##### 팀 템플릿

**리서치팀** (4명)
| 역할 | 이름(예시) | 모델 | 주요 스킬 |
|------|----------|------|----------|
| 조율자/종합 | 팀장 | Opus | 종합, 판단, 보고 |
| 탐색자/리서처 | 리서처 | Sonnet | web_search, web_fetch, summarize |
| 생산자/작성 | 라이터 | Sonnet | 콘텐츠 작성, 구조화 |
| 검증자/팩트체크 | 비평가 | Sonnet | 팩트체크, 논리 검증 |

**개발팀** (4명)
| 역할 | 이름(예시) | 모델 | 주요 스킬 |
|------|----------|------|----------|
| 조율자/PM | 팀장 | Opus | 요구사항 분석, 작업 분배 |
| 생산자/개발 | 개발자 | Sonnet | coding-agent, github, exec |
| 검증자/리뷰 | 리뷰어 | Sonnet | 코드 리뷰, 테스트 |

**콘텐츠팀** (5명)
| 역할 | 이름(예시) | 모델 | 주요 스킬 |
|------|----------|------|----------|
| 조율자/편집장 | 팀장 | Opus | 편집, 방향 설정 |
| 탐색자/기획 | 기획자 | Sonnet | 트렌드 리서치, 주제 발굴 |
| 생산자/작성 | 라이터 | Sonnet | 콘텐츠 작성 |
| 검증자/에디터 | 에디터 | Sonnet | 교정, 팩트체크, SEO |

**분석팀** (3명)
| 역할 | 이름(예시) | 모델 | 주요 스킬 |
|------|----------|------|----------|

#### Phase 4: 상호작용 설계

##### 기본 상호작용 패턴
```
[사용자 요청]
    → 조율자가 수신
    → 논점/작업 분해
    → 팀원들에게 분배
    → 팀원들이 각자 작업
    → 팀원 간 교차 검토/반론
    → 조율자가 종합
    → 사용자에게 보고
```

##### 상호작용 규칙 템플릿
1. 의견 제시 시 근거 필수
2. 반박 시 대안 함께 제시
3. 최대 3라운드 토론
4. 조율자가 합의 선언 시 종료
5. 팀원 간 직접 소통 가능

#### Phase 5: 테스트 & 검증
- 각 에이전트가 올바르게 로드되는지 확인
- 그룹 채팅에서 멘션 반응 테스트
- sessions_send 통신 테스트
- 역할에 맞는 응답 생성 확인
- 상호작용 패턴 동작 확인
```

## "매력적인 팀"이란 무엇인가

### 세나의 관점 (조율자)
매력적인 팀은 **각자 강점이 다르고, 그 강점이 시너지를 만드는 팀**이다.
- 미루의 빠른 리서치 + 유리의 날카로운 검증 = 신뢰할 수 있는 정보
- 하나의 실용적 구현 + 세나의 종합 판단 = 실행 가능한 결론
- 겉으로는 쿨한데 속으로 챙기는 관계 → 팀워크의 깊이

### 기능적 매력
1. **역할 차별화**: 각 팀원이 하는 일이 명확히 다르다
2. **교차 검증**: 한 사람의 결과를 다른 사람이 검증한다
3. **모델 다양화**: 비용 효율적 (Opus + Sonnet + Haiku)
4. **스킬 특화**: 각 팀원에게 다른 스킬셋 부여
5. **확장성**: 새 팀원 추가가 쉽다

### 상호작용 매력
1. **자연스러운 대화**: 기술적 호출이 아닌 이름 부르기
2. **캐릭터 깊이**: 나이, 성격, 관계 설정으로 개성 부여
3. **감정적 연결**: 팀원 간 관계 (언니/동생, 친구, 선후배)
4. **토론 문화**: 반박과 보완이 자연스러운 팀
5. **성장 서사**: 시간이 지나면서 관계가 깊어지는 설정

### 사용자 경험 매력
1. **투명성**: 팀원들의 토론 과정이 보인다
2. **참여감**: 사용자가 토론에 끼어들 수 있다
3. **신뢰감**: 여러 관점에서 검증된 결과
4. **재미**: 캐릭터 간 상호작용이 즐겁다
5. **효율성**: 한 명이 하는 것보다 빠르고 깊다

## 팀 매력 개선 방안

### 기능적 개선
1. **스킬 핫스왑**: 작업에 따라 팀원의 스킬셋을 동적으로 변경
2. **역할 로테이션**: 가끔 역할을 바꿔서 새로운 관점 도입
3. **팀 메모리**: 공유 MEMORY.md로 팀 차원의 학습
4. **자동 작업 감지**: 사용자 요청을 분석해서 최적 팀원 자동 배정
5. **성능 대시보드**: 각 팀원의 작업 기여도, 응답 품질 추적

### 상호작용 개선
1. **자발적 개입**: 다른 팀원의 작업에 자발적으로 피드백
2. **감정 반응**: 좋은 결과에 칭찬, 실수에 건설적 비판
3. **내부 토론**: 사용자 없이도 팀원끼리 품질 검토
4. **역사 참조**: 이전 작업 결과를 참조하여 개선
5. **팀 회고**: 작업 후 무엇이 잘됐고 무엇을 개선할지 논의

## 팀 빌더 사용 예시

### 예시 1: "SEO 콘텐츠팀 만들어줘"
```
사용자: "SEO 블로그 콘텐츠팀 만들어줘. 캐릭터형으로."

팀 빌더 결과:
├── AGENTS.md (팀 구조)
├── agents/
│   ├── editor/
│   │   └── SOUL.md (편집장 - 28세, 꼼꼼한 완벽주의자)
│   ├── writer/
│   │   └── SOUL.md (라이터 - 25세, 창의적 자유영혼)
│   ├── seo-specialist/
│   │   └── SOUL.md (SEO 전문가 - 30세, 데이터 기반 분석가)
│   └── researcher/
│       └── SOUL.md (리서처 - 22세, 호기심 많은 탐구자)
└── openclaw.json (설정 패치)
```

### 예시 2: "풀스택 개발팀 만들어줘"
```
사용자: "React + Node.js 풀스택 개발팀. 전문적 톤."

팀 빌더 결과:
├── AGENTS.md
├── agents/
│   ├── tech-lead/ (기술 리더 - 아키텍처 설계)
│   ├── frontend/ (프론트엔드 - React/Next.js)
│   ├── backend/ (백엔드 - Node.js/DB)
│   └── qa/ (QA - 테스트/코드 리뷰)
└── openclaw.json
```
