# Socratic Interview: 소크라틱 인터뷰 모듈

**버전**: 7.0 | **목적**: 옵션 제시 전 사용자 요구사항 구체화

---

## 개요

소크라틱 인터뷰는 **옵션 메뉴 제시 전에** 사용자가 원하는 것을 스스로 정리할 수 있도록 돕는 대화형 탐구 과정입니다.

**문제 인식:**
- 사용자는 종종 "뭘 원하는지 정리가 안 된" 상태로 요청
- 옵션 메뉴를 바로 보여주면 → 막막함 → 대충 선택 → 결과물 불만족
- 결과물 받고 나서야 "이게 아니었는데..." 깨달음

**해결책:**
- 5 Whys 기반 질문으로 요구사항 구체화
- 현재 워크플로우 파악 → 문제점 도출 → 이상 상태 정의
- 빌드 전 입력/출력 예시 + 시나리오 확인

---

## 핵심 원칙

### 1. 컨설턴트형 스타일

```markdown
# 나쁜 예 (수동적)
"어떤 스킬을 만들까요?"

# 좋은 예 (적극적 탐구)
"이 작업을 지금은 어떻게 하고 계세요?"
```

### 2. 5 Whys 깊이

표면적 요청이 아닌 **근본 문제**까지 파고들기:

```
[Level 1] 현재 상태 파악
    ↓
[Level 2] 문제점/불편함 도출
    ↓
[Level 3] 이상적 상태 정의
    ↓
[Level 4] 엣지케이스 탐색
    ↓
[Level 5] 요약 및 확인
```

### 3. 라이브 데모 효과

질문에 답하는 과정 자체가 사용자의 생각 정리를 도움:
- 현재 워크플로우 설명 → 뭘 원하는지 정리됨
- 불편한 점 말하기 → 핵심 문제 드러남
- 시나리오 확인 → Claude가 맥락 이해

## 인터뷰 흐름

### Phase 1: 현재 상태 파악 (Q1)

```markdown
🎯 **첫 번째 질문**

> "이 작업을 **지금은 어떻게** 하고 계세요?"

목적:
- 현재 워크플로우 파악
- 수동 작업 vs 부분 자동화 확인
- 사용 중인 도구/서비스 파악

예시 응답 유도:
- "수동으로 하나씩..."
- "엑셀로 정리하다가..."
- "다른 스킬/도구 쓰는데 부족해서..."
```

### Phase 2: 문제점 도출 (Q2)

```markdown
🎯 **두 번째 질문**

> "그중에서 **가장 짜증나는/시간 잡아먹는** 부분이 뭐예요?"

목적:
- 핵심 페인포인트 식별
- 우선순위 파악
- 자동화 ROI 높은 지점 확인

후속 질문 (선택):
- "그게 얼마나 자주 발생해요?"
- "한 번 발생하면 얼마나 걸려요?"
- "실수하면 어떤 일이 생겨요?"
```

### Phase 3: 이상적 상태 정의 (Q3)

```markdown
🎯 **세 번째 질문**

> "완벽하게 작동하면 **어떤 상태**가 돼요?"
> "**입력**이 뭐고 **출력**이 뭐면 좋겠어요?"

목적:
- 성공 기준 정의
- 구체적 입력/출력 예시 확보
- 기대치 명확화

예시 유도:
- "~를 넣으면 ~가 나오면 좋겠어요"
- "매일 아침 자동으로 ~가 정리돼 있으면..."
- "버튼 하나로 ~까지 끝나면..."
```

### Phase 4: 엣지케이스 탐색 (Q4)

```markdown
🎯 **네 번째 질문**

> "이런 경우는 어떻게 하면 돼요?"

시나리오 체크리스트:
- **예외 케이스**: "입력이 비어있으면?"
- **실패 케이스**: "API 연결 실패하면?"
- **경계 케이스**: "데이터가 1000개 넘으면?"
- **권한 케이스**: "다른 사람이 사용하면?"

목적:
- 빌드 후 "이 경우는?" 방지
- 요구사항 빈틈 채우기
- 실패 처리 방식 결정
```

### Phase 5: 요약 및 확인 (Q5)

```markdown
🎯 **다섯 번째 질문**

> "제가 이해한 게 맞는지 확인할게요:"

요약 포맷:
┌─────────────────────────────────────┐
│ 📋 **요구사항 요약**                   │
├─────────────────────────────────────┤
│ **현재 상태**: {현재 워크플로우}         │
│ **핵심 문제**: {가장 불편한 점}         │
│ **원하는 상태**: {이상적 결과}          │
│ **입력 예시**: {구체적 입력}            │
│ **출력 예시**: {구체적 출력}            │
│ **예외 처리**: {엣지케이스 대응}         │
└─────────────────────────────────────┘

> "이게 맞아요? 수정할 부분 있어요?"

확인 후 → 옵션 메뉴로 진행
```

## 인터뷰 트리거 조건

### 자동 트리거 (소크라틱 인터뷰 시작)

| 조건 | 판단 근거 |
|------|----------|
| 막연한 요청 | "~하는 스킬 만들어줘" (구체적 스펙 없음) |
| 문제 중심 요청 | "~가 불편해", "~를 자동화하고 싶어" |
| 질문형 요청 | "~하는 게 가능해?", "~하려면 어떻게?" |
| 첫 스킬 생성 | 사용자 히스토리에 스킬 생성 경험 없음 |

### 스킵 트리거 (바로 옵션 메뉴)

| 조건 | 판단 근거 |
|------|----------|
| 구체적 스펙 | 입력/출력/로직이 명확히 기술됨 |
| 레퍼런스 제공 | "이 스킬처럼", "이거 수정해서" |
| 명시적 스킵 | "바로 만들어", "질문 없이 진행" |
| 수정 요청 | 기존 스킬 수정 (맥락 이미 있음) |

## 적응형 질문 전략

### 응답 길이에 따른 조절

```python
if len(user_response) < 20:  # 짧은 응답
    # 후속 질문으로 더 파고들기
    ask_followup("조금 더 구체적으로 설명해주실 수 있어요?")
    
elif len(user_response) > 200:  # 긴 응답
    # 핵심 추출 후 확인
    summarize_and_confirm("핵심은 ~인 것 같은데 맞아요?")
    
else:  # 적절한 응답
    # 다음 단계로 진행
    proceed_to_next_phase()
```

### 불확실성 처리

```markdown
# 사용자: "잘 모르겠어요"

응답 전략:
1. 선택지 제공: "보통 이런 경우가 많아요: A, B, C 중에 가까운 게 있어요?"
2. 예시 제공: "예를 들면 ~같은 상황인가요?"
3. 스킵 허용: "일단 넘어가고 나중에 정해도 돼요"
```

### 다중 선택 처리

```markdown
# 사용자: "A랑 B 둘 다요" 또는 "A,B,C"

응답 전략:
1. 우선순위 확인: "둘 다 중요하면, 더 급한 건 뭐예요?"
2. 통합 가능 여부: "둘을 합쳐서 처리할 수도 있어요"
3. 분리 제안: "두 개 스킬로 나누는 게 나을 수도 있어요"
```

## 인터뷰 예시

### 예시 1: 막연한 요청

```markdown
👤 사용자: 이메일 관련 스킬 만들어줘

🤖 Claude: 
이메일 관련 스킬이요! 몇 가지 여쭤볼게요.

**Q1. 지금은 이메일을 어떻게 관리하고 계세요?**
(예: 수동 분류, 필터 사용, 다 읽기 포기 등)

👤 사용자: 중요한 거 놓칠까봐 다 읽는데 시간이 너무 걸려요

🤖 Claude:
아, 다 읽느라 시간이 많이 드는군요.

**Q2. 그중에서 가장 시간 잡아먹는 게 뭐예요?**
- 읽을 필요 없는 메일 걸러내기?
- 중요도 판단하기?
- 회신 작성하기?

👤 사용자: 읽을 필요 없는 거 걸러내는 게 제일 오래 걸려요

🤖 Claude:
**Q3. 완벽하게 작동하면 어떤 상태가 돼요?**

예를 들면:
- "읽어야 할 메일만 받은편지함에 남아있다"
- "중요도 태그가 자동으로 붙는다"
- "요약이 먼저 보인다"

👤 사용자: 읽어야 할 것만 남아있으면 좋겠어요

🤖 Claude:
**Q4. 이런 경우는 어떻게 할까요?**
- 뉴스레터는? (읽을 때도 있고 아닐 때도?)
- 참조(CC) 메일은?
- 처음 보는 발신자는?

👤 사용자: 뉴스레터는 따로 폴더로, CC는 나중에 볼 폴더로, 처음 보는 건 일단 남겨둬요

🤖 Claude:
**Q5. 정리해볼게요:**

┌─────────────────────────────────────┐
│ 📋 **요구사항 요약**                   │
├─────────────────────────────────────┤
│ **현재**: 모든 메일 수동 확인          │
│ **문제**: 불필요 메일 걸러내기 시간 소요 │
│ **원하는 상태**: 읽어야 할 메일만 표시   │
│ **규칙**:                            │
│  - 뉴스레터 → 별도 폴더              │
│  - CC 메일 → "나중에" 폴더           │
│  - 신규 발신자 → 받은편지함 유지       │
└─────────────────────────────────────┘

이게 맞아요? → [확인] → 옵션 메뉴로 진행
```

## 구현 체크리스트

### Claude 구현 시 확인사항

**인터뷰 시작:**
- [ ] 트리거 조건 확인 (막연한 요청?)
- [ ] Q1부터 순차 진행
- [ ] 컨설턴트형 톤 유지

**인터뷰 진행:**
- [ ] 응답 길이에 따라 후속 질문 조절
- [ ] "모르겠다"에 선택지 제공
- [ ] 다중 선택 시 우선순위 확인

**인터뷰 종료:**
- [ ] 요약 포맷으로 정리
- [ ] 확인 질문 ("이게 맞아요?")
- [ ] 확인 후 옵션 메뉴로 이동

**스킵 처리:**
- [ ] 명시적 스킵 요청 존중
- [ ] 구체적 스펙 있으면 바로 진행
- [ ] 스킵해도 핵심 질문 1개는 확인
